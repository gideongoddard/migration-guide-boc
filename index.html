Website handover

Brixworth Osteopathic Clinic — transferring ownership

A step-by-step guide for transferring the site, its hosting, and its code repository into your ownership.

Who this guide is for

This document is for the developer taking over maintenance of the Brixworth Osteopathic Clinic website. It covers the one-time transfer of ownership only — for ongoing reference about how the site is built and maintained, see the README.md in the code repository.

Work through the steps in order. Some of them need to happen at specific points relative to others, so skipping ahead can cause brief outages.

Context for the rest of the guide

The website is built with Gatsby (a React-based static site generator), hosted on Netlify, with the code in a private GitHub repository. The client can edit blog articles through an admin interface at /admin on the live site, powered by Netlify CMS. Contact form submissions go through Netlify Forms and arrive by email.

There is no database. Each time content is updated, Netlify rebuilds the entire site from the code and publishes the new version. This takes around a minute or two per build.

ℹ
The README has the detail For a full explanation of the stack, how it all fits together, how to run it locally, and how to maintain it, read README.md in the repository. This guide focuses only on getting ownership transferred cleanly.
!
One constraint worth knowing upfront The CMS authentication uses Netlify Identity, a service Netlify has deprecated (existing instances still work, but no new ones can be created). This means the Netlify site transfer feature is the only clean handover path. Rebuilding the site from scratch on a fresh Netlify account would leave the CMS unable to authenticate. The steps below are sequenced to avoid this.
The handover, step by step
1
Create the accounts you'll need
You do this — 5 to 10 minutes You do this

You'll need a GitHub account and a Netlify account. If you already have personal ones you're comfortable using for this, they're fine. Otherwise, create them now.

  • GitHub account Sign up at github.com/signup. A free personal account is sufficient. Make a note of the username — you'll need to share it in step 2.
  • Netlify account Sign up at app.netlify.com/signup. You'll be prompted to create a team — a free Starter team is fine. Sign up using your GitHub account if you want; it makes later steps slightly smoother.
  • Share both with Gideon He'll need your GitHub username (for the repo transfer) and the email address on your Netlify account (for the site transfer).
2
Receive the GitHub repository
You do this — a few minutes You do this

Gideon will initiate a repository transfer from his GitHub account to yours. You'll receive an email from GitHub asking you to accept.

  • Accept the transfer Open the email from GitHub and click the link to accept. Alternatively, log in to GitHub and look for the pending transfer notification at the top of your dashboard.
  • Confirm the repo appears in your account The repository should now be at https://github.com/<your-username>/brixworth-osteo. All commit history, branches, and settings come with it.
ℹ
The repo name stays as brixworth-osteo The CMS config file inside the repo references this name. If you rename the repo later, you'll need to update static/admin/config.yml to match — the README covers this.
3
Receive the Netlify site
You do this — a few minutes You do this

Gideon will transfer the Netlify site to your Netlify team. You'll receive an email and an in-app notification.

  • Accept the transfer in Netlify Log in to Netlify and look for the pending transfer notification, or follow the link in the email. Confirm acceptance.
  • What moves across automatically Deploy history, the live site, custom domain, DNS zone, Netlify Forms submissions, form notification settings, Netlify Identity users (the client and their assistant), Git Gateway configuration, and build settings. Essentially, the site continues running exactly as it did, just under your account.
  • One thing that doesn't move The connection between the Netlify site and the GitHub repository. That's reconnected in the next step. Until then, the site is still live — new commits just won't trigger builds.
✓
No DNS changes are needed The domain is already using Netlify DNS, and the nameservers continue to point to Netlify throughout — only the underlying team ownership changes. The site stays live the entire time.
4
Reconnect the GitHub integration
You do this — a few minutes You do this

Netlify needs permission from your GitHub account to read the repository and trigger builds when you push commits. This permission is user-specific, so it needs re-authorising under your account.

  • Open the site's build settings in Netlify In the transferred site, go to Site configuration → Build & deploy → Continuous deployment. You should see a prompt to reconnect or reauthorise GitHub access.
  • Authorise Netlify on GitHub Follow the prompts. You'll be bounced to GitHub to approve the Netlify app for your account and grant it access to the brixworth-osteo repository. Approve it.
  • Trigger a test build Back in Netlify, go to Deploys and click Trigger deploy → Deploy site. Watch the build log. If it completes successfully, the integration is working.
5
Set up a new GitHub OAuth app for the CMS
You do this — 10 minutes You do this

The CMS admin login offers "Log in with GitHub" as an option. That flow depends on a GitHub OAuth app — currently one that belongs to Gideon. You'll create a fresh one under your account and swap it in, so nothing is left tied to him.

!
Brief CMS login disruption during this step Between disabling the old OAuth app and enabling the new one, the "Log in with GitHub" option in the CMS will not work. The public website is unaffected. Plan to do this when the client is not trying to edit content.
  • Create a new OAuth app on GitHub Go to GitHub → Settings → Developer settings → OAuth Apps and click New OAuth App.
  • Fill in the OAuth app details Application name: something like Brixworth Osteopathic Clinic CMS. Homepage URL: the site's live URL. Authorization callback URL: https://api.netlify.com/auth/done (this is a Netlify endpoint — don't change it). Click Register application.
  • Generate a client secret On the app's page, click Generate a new client secret. Copy both the Client ID and the new Client Secret into a safe place — the secret will only be shown once.
  • Paste them into Netlify Identity settings In the Netlify site, go to Integrations → Identity → Settings and usage → External providers. Find the existing GitHub provider and edit it. Replace the Client ID and Client Secret with the new values. Save.
  • Test the CMS login Go to https://<the-site>/admin in a private browser window. Click Log in with GitHub, authorise the new OAuth app when prompted, and confirm the CMS loads. If it does, the swap worked.
  • Let Gideon know Once verified, tell Gideon that the swap is complete. He'll delete the old OAuth app from his GitHub account.
ℹ
Why the callback URL is a Netlify address Netlify acts as an intermediary for the OAuth flow so that the CMS doesn't need its own backend server. This is standard for Netlify-hosted CMS setups — api.netlify.com/auth/done is shared by all Netlify sites using GitHub OAuth.
6
Verify everything works
You do this — 15 minutes You do this

Before calling the handover complete, run through this checklist end-to-end. If anything fails, stop and troubleshoot before moving on.

  • The live site loads Visit the site in a browser. Check the homepage, a few internal pages, and an existing blog article. Everything should render normally.
  • A code change triggers a build Clone the repository locally (or edit a file directly on GitHub), make a trivial change like updating a comment, commit, and push. Watch the Netlify Deploys tab — a new deploy should start within seconds and complete successfully.
  • CMS login via GitHub works Already tested in step 5, but worth confirming again from a fresh browser session.
  • Existing CMS users can still log in Ask the client to log in to /admin using their existing Netlify Identity email credentials. They should be able to edit an article and publish it successfully. The publish action should trigger a new Netlify build.
  • Contact form submission arrives Submit the contact form on the live site with test content. Confirm the submission appears in Netlify under Forms, and that the notification email reaches the client's email address as expected.
  • Domain resolves correctly The domain should continue to point to the site with a valid SSL certificate. Check Domain management in Netlify — the custom domain should be listed as primary, with HTTPS enabled and a valid certificate.
✓
If everything above passes, the handover is complete You now fully own the site, the code, the hosting, the CMS, and the OAuth integration. Nothing is left tied to Gideon's accounts.

Ongoing responsibilities
What's now on your plate
  • Hosting costs Netlify's Starter plan is free and handles this site's traffic comfortably. If the client's traffic grows or additional features are needed, you may need to upgrade.
  • Domain renewal The domain is registered separately at its registrar — ask the client for those details. Renewal reminders usually come by email to whoever's on the registrar account.
  • Security of your accounts Enable two-factor authentication on both GitHub and Netlify. Both the site's source code and its hosting are now accessible through those accounts.
  • Ongoing maintenance The README documents the current state of the stack, including several dependencies that are due for upgrade. Work through those at a pace you're comfortable with. None are urgent but they'll become more pressing over time.
